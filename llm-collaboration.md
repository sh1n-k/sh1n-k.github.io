# LLM 활용 방법론

이 과제는 LLM을 “답을 대신 쓰는 도구”가 아니라, 불확실한 요구사항을 구조화하고 설계 결정을 빠르게 검증하는 **협업 파트너**로 사용했다. 최종 산출물(가정 세트, 서버 작업 계획, 관리자 화면 설계)은 LLM이 생성한 초안을 기반으로 했지만, **문제 분해·가정 확정·트레이드오프 선택·리스크 통제**는 사람이 책임지고 결정한다는 원칙을 유지했다.

## **목표와 원칙**

첫째, **평가 포인트 정렬**이 목표였다. 과제는 “LLM을 적극 권장하며, 어떤 프롬프트로 어떤 결과를 의도했는지”와 “사고 결정과 의사결정 근거”를 요구한다. 따라서 산출물 자체뿐 아니라, 산출물에 도달하기 위한 **질문 설계와 선택 근거**가 문서의 핵심이 된다.

둘째, **가정을 먼저 고정**했다. 설계 과제에서 가장 흔한 실패는 애매한 요구사항을 그대로 구현 계획으로 끌고 가는 것이다. 그래서 LLM에게도 먼저 “필수 가정 사항을 정리”하도록 요청해, 문제의 전제를 명시화하는 단계부터 시작했다.

셋째, **산출물은 구조화된 포맷으로 강제**했다. “작업 단계 / 기술적 결정 사항 / 리스크 및 대응” 같이 평가자가 읽기 쉬운 형태로 결과를 만들도록 프롬프트에 출력 구조를 포함했다.

## **실제 사용 흐름 (의사결정 중심)**

전체 흐름은 다음과 같이 설계했다.

1) **가정 수립(불확실성 제거)**  
- 의도: 무엇이 확정이고 무엇이 가정인지 먼저 분리해 설계의 ‘기준선’을 만든다.  
- 결과물: “업체 선택 단위, API 통신 방식, 인증정보 저장, 호환성 유지 범위, Webhook 처리 방식” 등 필수 전제의 목록화.

2) **API 차이점 기반 설계 영향 도출**  
- 의도: 업체별 차이(인증, 엔드포인트, 전화번호 포맷, 필드명)가 실제 구현에 어떤 변화를 강제하는지 추려낸다.  
- 결과물: Provider 추상화 필요성, 번호 포맷 정규화, Webhook 분리 처리, 내부 도메인 모델의 중립 필드 정의.

3) **최종 가정 세트 확정(제출 가능한 형태로 정리)**  
- 의도: 추가 정보가 없더라도 실무적으로 타당한 가정 세트를 만들고, 그 위에서 설계를 완결한다.  
- 결과물: “작업 범위/업체 선택 정책/인증 관리/매핑 관리/호환성/Webhook/데이터 모델/에러·Fallback” 8개 항목 정리.

4) **서버 측 작업 계획(실행 가능 단위로 분해)**  
- 의도: ‘무엇을 먼저/왜’가 드러나도록 마이그레이션부터 배포까지 순서를 만든다.  
- 결과물: 데이터 모델 확장 → Provider 추상화 → Resolver → 발송 서비스 리팩토링 → Webhook → 관리 API → 테스트 → 배포.

5) **요구사항 변경 반영(가정 업데이트 후 재설계)**  
- 의도: 두 번째 문제(관리자 화면)가 기존 전제를 뒤집는지 먼저 점검하고, 일관성 있게 수정한다.  
- 결과물: “관리자가 화면에서 업체를 설정”하도록 가정 변경, 관리 API 확장(연결 테스트/검증/마스킹/캐시 무효화/변경 이력/낙관적 락).

6) **관리자 화면 설계(실수 방지 중심)**  
- 의도: 운영자 실수를 전제로, 입력 오류를 사전에 줄이고 변경 영향도를 명확히 전달하는 UX를 만든다.  
- 결과물: 조직 선택 → 업체 선택 → 인증 입력 → 연결 테스트/저장 흐름, 동적 폼·테스트 필수화·확인 모달·삭제 보호·충돌 처리.

## **프롬프트 설계(의도 → 기대 출력)**

과제에서 요구하는 “프롬프트 의도”는 다음과 같이 정리할 수 있다.

- “가정을 명시적으로 정의… 필수 가정 사항 정리”  
  - 의도: 불확실성을 제거하고 설계 기준선을 만든다.  
  - 기대 출력: 작업 범위/정책/호환성/데이터 모델/운영 권한 등 결정해야 할 전제 리스트.

- “API 명세 참조해 가정 정리 + 더 필요한 정보 요구”  
  - 의도: 스펙 차이점이 설계에 미치는 영향을 뽑고, 진짜로 필요한 추가 정보를 드러낸다.  
  - 기대 출력: 인증/엔드포인트/포맷/필드 차이와 설계 영향, 그리고 부족한 정보(기존 API/이력 스키마/권한/보안정책) 목록.

- “추가 정보는 네가 적절하게 정의. 최종 가정 세트 작성”  
  - 의도: 현실적으로 정보가 불완전해도 ‘제출 가능한’ 일관된 답안을 완결한다.  
  - 기대 출력: 명시적 가정 + 근거 + 배제 범위가 포함된 가정 세트.

- “서버 측 작업 계획을 카테고리대로”  
  - 의도: 실행 계획과 의사결정 포인트, 리스크 통제가 평가자 관점에서 한눈에 보이게 한다.  
  - 기대 출력: 단계/결정/리스크 3축 구조의 답변.

- “두 번째 문제로 가정 변경 필요 여부 판단 후 재답변”  
  - 의도: 요구사항 변화가 전제와 설계를 어떻게 바꾸는지 일관되게 보여준다.  
  - 기대 출력: 변경된 가정 + 변경된 서버 계획 + 추가된 UI 설계.

## **검토·채택 기준(내가 책임지는 부분)**

LLM 출력은 다음 기준으로 검토하고 채택/수정한다.

- **요구사항 적합성**: 문제에서 요구한 범위(SMS/MMS/LMS/알림톡, 앱푸시 제외), 운영자가 변경 가능해야 한다는 조건 반영 여부.
- **일관성**: 가정 세트 ↔ 서버 설계 ↔ UI 설계가 서로 충돌하지 않는지.
- **실행 가능성**: 실제 개발 단계로 쪼개졌는지(마이그레이션, API, 테스트, 배포 순서).
- **운영 안전성**: 실수 방지 장치(연결 테스트, 마스킹, 변경 이력, 충돌 방지)가 있는지.
- **보안 기본기**: 인증정보 저장/조회/마스킹/키 관리 방향이 최소 기준을 충족하는지.

채택 이후에도, “추상화가 과한가/부족한가”, “연결 테스트를 필수화하면 운영이 막히지 않는가” 같은 트레이드오프는 최종 문서에서 **명시적으로 근거**를 남기는 방식으로 처리한다.

## **중학생 수준 비유(왜 이런 흐름이 자연스러운가)**

두 업체를 바꿔 끼우는 문제는, ‘충전기’를 바꾸는 것과 비슷하다.

- 휴대폰(우리 서비스)은 그대로인데,  
- 충전기(메시지 업체)가 2종류로 늘어난다.  
- 그러면 휴대폰에는 “충전기 꽂는 규격(공통 인터페이스)”이 필요하고,  
- 각 충전기는 “어댑터(업체별 구현)”로 맞춰서 꽂아야 한다.  
- 운영자가 충전기를 바꾸는 순간(관리 화면 변경) 실수하면 폰이 충전이 안 되니,  
  **꽂기 전에 테스트(연결 테스트)**하고, **확인(모달)**하고, **누가 언제 바꿨는지 기록(변경 이력)**하는 게 안전하다.

## **한계와 보완 계획(정직한 범위 표시)**

- 실제 API 문서 원문이 공개되지 않은 상태에서, “Webhook 지원/번호 포맷/필드명 차이” 등은 과제 대화에 포함된 전제를 기반으로 정리했다.  
- 실제 제출 시에는 API 문서 링크/발췌(허용 범위 내)와 함께, “문서에서 확인한 사실”과 “내가 둔 가정”을 명확히 분리 표기하는 것이 안전하다.
