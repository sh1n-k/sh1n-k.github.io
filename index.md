# 과제 정리: 메시지 발송 업체 멀티 프로바이더 설계 + 관리자 설정 화면

이 문서는 “앱 푸시를 제외한 메시지(SMS, MMS/LMS, 알림톡)”를 기존 **CloudMsg** 기반에서, 조직(Organization)별로 **CloudMsg 또는 SendTalk**을 선택해 발송할 수 있도록 확장하는 설계를 정리한 결과물입니다.  
추가로, 서비스 운용자가 조직별 업체를 변경할 수 있는 **관리자 화면 설계**를 포함합니다.

본 과제는 기업의 요구에 맞춰 **LLM을 도구로 활용한 문제 해결 과정(프롬프트 의도·산출물·의사결정 근거)**까지 함께 기록합니다.

---

## 목차 (Pages)

1) **LLM 활용 방법론** (`/llm-collaboration`)  
- 어떤 프롬프트로 무엇을 의도했고, 어떤 기준으로 채택/수정했는지  
- 최종 산출물의 “결정 흐름”과 책임 범위 정리

2) **문제 분석** (`/problem-analysis`)  
- 요구사항 요약 및 범위 정의  
- 과제 해결을 위한 문제 분해(하위 질문) 구조

3) **API 비교 분석** (`/api-comparison`)  
- CloudMsg vs SendTalk: 인증/엔드포인트/필드 차이  
- 설계에 영향을 주는 핵심 차이와 처리 전략

4) **가정 세트** (`/assumptions`)  
- 필수 전제(범위·업체 선택 단위·보안·호환성·Webhook 등)  
- 각 가정의 근거와 트레이드오프

5) **서버 설계 및 작업 계획** (`/server-design`)  
- 작업 단계(마이그레이션→추상화→Webhook→관리 API→테스트→배포)  
- 기술적 결정 사항(Provider 인터페이스/스키마/캐시/트랜잭션)  
- 리스크 및 대응(회귀/유실/운영 실수/보안)

6) **관리자 화면 설계** (`/admin-ui-design`)  
- 화면 흐름(조직 선택→업체 선택→인증 입력→연결 테스트/저장)  
- UI 결정 사항(동적 폼/테스트 필수화/마스킹/확인 절차)  
- 예외/엣지 케이스 대응(동시 수정/삭제 복구/테스트 장애 등)

7) **부록** (`/appendix`)  
- 참고 자료(가능 시 API 문서 링크/요약)  
- 대화 로그 요약(핵심 의사결정 포인트 중심)

## 접근 방식 요약

핵심 전략은 다음 3단계입니다.

- **가정 명시**: 불완전한 요구사항을 그대로 설계로 끌고 가지 않고, 설계에 영향을 주는 전제를 먼저 고정합니다.  
- **추상화(Provider) 도입**: 발송 로직은 공통 인터페이스로 통일하고, 업체별 차이는 Adapter/Mapper로 격리합니다.  
- **운영 안전장치**: 관리자 화면 입력 실수와 변경 리스크를 줄이기 위해 연결 테스트, 마스킹, 변경 이력, 잠금/충돌 처리 등을 포함합니다.
