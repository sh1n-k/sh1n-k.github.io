# 부록

## **A. 문서/자료 구성 원칙**

- “사실(문서로 확인됨)”과 “가정(합리적 추론)”을 분리 표기하는 것을 원칙으로 한다.
- 이 레포는 최종 산출물뿐 아니라, **LLM을 활용한 사고 흐름**을 평가자가 빠르게 검토할 수 있도록 페이지를 분리했다.

## **B. 구현 시 참고할 API/데이터 체크리스트(요약)**

1) Provider 구현 체크
- 인증 방식 구현(HMAC/Token)
- 번호 포맷 변환(E.164 ↔ local)
- 알림톡 템플릿/변수/버튼 매핑
- 요청 실패 시 에러 표준화(내부 에러 코드/메시지)
- `validateCredentials()` 최소 호출 정의(가능하면 read-only 성격)

2) Webhook 구현 체크
- 서명 검증(업체별)
- 멱등 처리(idempotency key)
- 상태 전이 제한(허용되는 전이만)
- 수신 시각 기록
- 운영 추적용 requestId/log correlation

3) 관리자 운영 체크
- 마스킹 정책(원문 노출 금지)
- 변경 이력(Audit Log)
- 낙관적 락(버전 충돌)
- 캐시 무효화(변경 즉시 반영)

## **C. 대화 로그 요약(의사결정 포인트 중심)**

- 먼저 “필수 가정”을 명시화하여 설계 기준선을 만들었다.
- API 차이를 설계 영향(추상화/모델 표준화/Webhook 분리)로 연결했다.
- 두 번째 문제(관리자 화면)가 등장하자, 가정 중 “매핑 변경 주체/UI 필요” 전제가 바뀌므로 가정 세트와 서버 계획을 함께 갱신했다.
- 운영 실수 리스크를 핵심 리스크로 보고, 연결 테스트/확인 모달/변경 이력/충돌 방지 같은 안전장치를 설계에 포함했다.
